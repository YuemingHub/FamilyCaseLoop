# cases/ · 真实案例容器（S8 落点）

本目录是 **FamilyCaseLoop 同行版** 的「真实案例容器」，对应个案全生命周期链路 `S0–S8` 的最终回流收口节点（S8）。

> 引擎（`engine/`）负责 S3/S5/S6/S7 的分析与生成；引导前端（`guide/`）在 S8 提供「导出本次分析为个案草稿」与「下载填写模板」两个纯前端动作，把结果落回本目录。**本目录不依赖任何外部大仓路径，同行版完全 decoupled。**

---

## 目录结构

```
cases/
├── README.md              # 本说明（入库规矩）
├── _template/
│   └── CASE_TEMPLATE.md   # 单个个案全链填写模板（S0–S8）
├── _inbox/                # 原始材料临时存放（.gitignore 屏蔽，绝不进版本库）
│   └── {化名}/            #  每个个案一个子目录，放 zip/docx/录音等原始文件
└── active/                # 脱敏后的服务档案存放处 ← 可 push 到版本库
    └── 个案_{化名}_{季度}.md
```

---

## ⛔ 脱敏红线（铁律，不可协商）

**未脱敏的原始个案不得进入 `active/`、不得 push 到版本库、不得进入任何对外/同行链路。**

### 脱敏检查清单（每个个案入库前逐项过）

| # | 检查项 | 示例（来自甲鱼案实际处理） | 处理方式 |
|---|--------|--------------------------|---------|
| 1 | 真实姓名（家长/孩子/配偶/老师） | 张克将、戚娟丽、张芮齐 | 替换为化名（如「甲鱼妈」「甲鱼」「甲鱼爸」） |
| 2 | 电话号码 / 微信号 / QQ 号 | 139xxxxxxxx、wx: xxxxx | 完全删除，不保留任何形式 |
| 3 | 具体地名（小区/学校/城市） | 宿迁某中学 | 降级为「某市」「某校」或完全删除 |
| 4 | 可定位的组合线索 | 城市名+校名+年级+事件=可定位 | 至少打散 2 项，使外部无法通过交叉检索定位当事人 |
| 5 | 具体日期 / 证件号 / 门牌号 | 2024年X月X日 | 降级为时间段（如「去年底」「今年6月」） |

> **判断标准**：如果有人拿到这份档案 + 搜索引擎，能否定位到具体家庭？能 → 未通过，继续脱敏或只留本地。

## 入库决策树

```
收到原始个案材料（zip/docx/录音/聊天记录）
  │
  ▼
存入 cases/_inbox/{化名}/          ← .gitignore 屏蔽，绝不 push
  │
  ▼
提取文本 → 脱敏（按上方清单逐项过）
  │
  ▼
脱敏通过吗？
  ├─ 否 → 继续脱敏，或标记「仅本地」存 _inbox/，不入 active/
  │
  └─ 是 → 走 S0–S8 全流程（guide/ 或手动）
           │
           ▼
         S3 安全门通过吗？
           ├─ 否（高危） → 标记「高危·仅本地」，不入 active/
           │
           └─ 是 → 写脱敏档案到 active/个案_{化名}_{季度}.md
                    │
                    ▼
                  含强线索/可交叉定位吗？
                    ├─ 是 → 标记「敏感·仅本地」，留在 active/ 但不 push
                    │       （本地 git add 但不 push，或加 .gitignore 局部排除）
                    │
                    └─ 否 → ✅ 可 push 到版本库
```

### 三个级别的入库策略

| 级别 | 条件 | 存放位置 | 是否 push | 说明 |
|------|------|---------|----------|------|
| **🟢 可入库** | 脱敏通过 + 安全门常规 + 无强交叉线索 | `active/` | ✅ push | 标准个案，甲鱼案即此级别 |
| **🟡 敏感·仅本地** | 脱敏通过但仍有可交叉定位的线索组合 | `active/` | ❌ 不 push | 本地保留供复盘，不进公开仓库 |
| **🔴 高危·仅本地** | S3 安全门触发高危（自伤/伤人/被侵害） | `_inbox/` | ❌ 不 push | 原始材料级别，不脱敏进 active/ |

## 命名规范

```
active/个案_{化名}_{年份}Q{季度}.md
```

- 化名：用家长化名或家庭代号（如「甲鱼妈」「小雨家」），不用真实姓氏
- 季度：`2026Q3` = 2026年第三季度
- 同一家庭多个个案：`个案_甲鱼妈_2026Q3_02.md`（末尾加序号）

## 入库操作流程（基于甲鱼案验证）

1. **收材料** → 解压/复制到 `cases/_inbox/{化名}/`（已被 `.gitignore` 屏蔽）
2. **提取文本** → 从 docx/录音文稿提取纯文本，存 `_inbox/{化名}/case_text.txt`
3. **脱敏** → 按脱敏检查清单逐项过，产出脱敏后的家长自述文本
4. **走全流程** → 在 `guide/` 或手动跑引擎，完成 S0–S8
5. **写档案** → 按 `_template/CASE_TEMPLATE.md` 结构写脱敏档案，存 `active/`
6. **自检** → 复读一遍，确认没有残留真实信息；确认 S3 安全门结论
7. **定级别** → 按入库决策树判断是 🟢/🟡/🔴
8. **入库** → 🟢 可 `git add + commit + push`；🟡/🔴 只留本地

---

## 提交约定

- `active/` 下 🟢 级个案正常 `git add` + `commit` + `push`。
- 🟡/🔴 级个案：本地保留，不 push。如需版本管理，在本地 commit 但不 push（`git commit` 后不执行 `push`）。
- `README.md` / `_template/` 随仓库版本走，正常提交。
- `_inbox/` 永远不进版本库（`.gitignore` 已屏蔽）。

## 与映射文档的关系

- `docs/case-lifecycle-mapping.md` 里 S8 的「缺口①」即指本容器，现已补建。
- 本容器只存**脱敏后的服务档案**，不存原始录音/文稿；原始材料归 `_inbox/`（本地临时存放，不进库）。
